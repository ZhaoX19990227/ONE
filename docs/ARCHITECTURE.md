# 技术架构

## 决策

首版采用低成本模块化单体：Java 21、Spring Boot、MySQL 8.4 和微信原生小程序。单台阿里云 2核4GB ECS 可运行，图片首版落本地 volume，后续通过存储接口平滑切 OSS。目录维护首期直接使用 SQL migration，暂不为一个微信群增加独立管理端。

## 后端模块

- `identity`：微信登录、用户资料、敏感字段可见性。
- `catalog`：吃饭品类、品牌、产品、别名和品牌标识。
- `recommendation`：上下文过滤、加权候选、选择与记忆解释。
- `record`：吃饭、饮品和鹿一下记录。
- `memory`：从反馈生成可追溯偏好事实。
- `analytics`：日历、日周月统计与金额聚合。
- `media`：图片元数据和未来 OSS 适配。
- 海报由小程序 Canvas 本地渲染，后端只给出脱敏后的聚合数据。

## 推荐流程

1. 根据维度、时间、预算和用餐方式过滤目录。
2. 计算时间适配、历史喜好、近期重复、探索和预算分数。
3. 查询与候选品牌、产品和属性相关的记忆。
4. 对甜度等可调整属性生成新的配置建议。
5. 返回三个候选、推荐理由和记忆引用。
6. 用户实际记录后生成新的记忆信号。

Qwen3.6 多模态负责图片识别并返回结构化候选；后端再通过品牌别名与目录做归一化。推荐排序即使模型不可用也能由本地记忆引擎完成。

## 隐私边界

`PRIVATE_HABIT` 使用独立明细表、独立接口和独立权限路径；不进入海报查询、群统计、管理端和模型请求。日志不得输出属性、身高体重、身体感受、微信 code、OpenID 和 Token。
