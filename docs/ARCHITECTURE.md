# ONE 技术架构

## 决策摘要

首版采用模块化单体，避免微服务、Kubernetes、独立搜索集群和强依赖 Redis。单台阿里云 ECS 即可运行 Caddy、Java 服务、管理端和 MySQL。

## 模块边界

- `identity`：微信登录、令牌、账号状态。
- `profile`：用户资料、兴趣护照、隐私可见性。
- `activity`：活动模板、生命周期、详情和推荐查询。
- `enrollment`：报名、审核、候补、签到和名额一致性。
- `trust`：举报、拉黑、内容审核和风险处置。
- `admin`：运营查询与处置接口。
- `integration`：微信、支付、AI、消息和对象存储适配器。

## 一致性策略

- MySQL 是活动名额和报名状态的唯一事实源。
- `activity_id + user_id` 唯一约束防止重复报名。
- 报名事务锁定活动记录，校验截止时间和名额后更新计数。
- 支付回调、外部通知和未来 AI 任务使用幂等键。
- 异步副作用写入 `outbox_event`，后台任务投递；规模增长后可替换为消息队列。
- Redis 未来仅用于热点缓存、限流和短期幂等，不承担名额扣减真相。

## 数据隐私

- OpenID、手机号、精确位置和实名信息分级存储。
- 首页只返回城市、区域和模糊距离。
- 生产日志禁止输出微信 code、OpenID、手机号和 Token。
- AI 请求经过脱敏，不发送精确地址、手机号和身份信息。

## 扩展路径

1. 单体多实例 + MySQL。
2. 增加 Redis 和对象存储。
3. 将通知、AI、内容审核拆成异步 Worker。
4. 数据规模达到瓶颈后引入搜索召回和独立推荐服务。
